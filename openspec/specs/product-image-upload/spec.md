## Requirements
### Requirement: Presigned URL Generation
The system SHALL generate AWS S3 presigned URLs for direct client-to-S3 image uploads. The backend SHALL validate file type and size before generating the presigned URL.

#### Scenario: Generate presigned URL for valid image
- **WHEN** frontend requests presigned URL with product-slug, file name, file type, and file size
- **AND** file type is jpg, jpeg, png, or webp
- **AND** file size is 5MB or less
- **THEN** backend returns presigned PUT URL with 15-minute expiration
- **AND** presigned URL includes Content-Type metadata matching file type

#### Scenario: Reject invalid file type
- **WHEN** frontend requests presigned URL with unsupported file type (e.g., pdf, exe)
- **THEN** backend returns 400 Bad Request with error message

#### Scenario: Reject file exceeding size limit
- **WHEN** frontend requests presigned URL with file size exceeding 5MB
- **THEN** backend returns 400 Bad Request with error message

### Requirement: S3 Folder Structure
All product images SHALL be stored in S3 using the folder structure `products/[product-slug]/`, where `product-slug` is a URL-safe slug generated from the product name.

#### Scenario: Image stored in correct folder
- **WHEN** image is uploaded for product with name "iPhone 15 Pro Max"
- **THEN** image is stored at S3 key `products/iphone-15-pro-max/[filename]`
- **AND** product-slug is generated by converting product name to lowercase, replacing spaces with hyphens, and removing special characters

### Requirement: Direct Client Upload
The system SHALL support direct browser-to-S3 uploads using presigned URLs, without proxying files through the backend.

#### Scenario: Successful direct upload
- **WHEN** frontend receives presigned URL from backend
- **AND** frontend uploads file directly to S3 using presigned URL
- **THEN** file is stored in S3 bucket
- **AND** frontend receives success response from S3
- **AND** frontend sends image metadata (S3 key, CloudFront URL) to backend to save in database

#### Scenario: Presigned URL expiration
- **WHEN** presigned URL expires before upload completes
- **THEN** frontend requests new presigned URL
- **AND** backend generates new presigned URL

### Requirement: Product Image Limits
Each product SHALL support a maximum of 4 images: 1 primary image and up to 3 secondary images.

#### Scenario: Upload primary image
- **WHEN** user uploads first image for a product
- **THEN** image is marked as primary (isPrimary=true)
- **AND** image is saved to database with productId

#### Scenario: Upload secondary images
- **WHEN** user uploads additional images (2nd, 3rd, 4th)
- **THEN** images are marked as secondary (isPrimary=false)
- **AND** images are saved with sortOrder (1, 2, 3)

#### Scenario: Reject exceeding image limit
- **WHEN** user attempts to upload 5th image for a product
- **THEN** system rejects upload with error message indicating maximum 4 images allowed

### Requirement: CloudFront URL Generation
All image URLs returned to frontend SHALL use CloudFront domain instead of direct S3 URLs.

#### Scenario: CloudFront URL in API response
- **WHEN** product images are retrieved from database
- **THEN** all image URLs are transformed to CloudFront format: `https://d1ffmiafbbgufv.cloudfront.net/[s3-key]`
- **AND** S3 key is stored in database, CloudFront URL is generated on read

#### Scenario: CloudFront URL for new upload
- **WHEN** image is uploaded and saved to database
- **THEN** CloudFront URL is generated and returned in API response
- **AND** database stores S3 key, not full CloudFront URL
